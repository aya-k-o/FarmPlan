# FarmPlan 要件定義書

> ポートフォリオ設計資料（2025年3月策定・実装後更新）

---

## 1. サービス概要

畑の区画をグリッドで可視化し、野菜の配置・連作障害チェック・来季のシミュレーションを支援するWebアプリ。

**コンセプト**
> 作付け計画を、悩ましい作業からワクワクする体験へ。

---

## 2. 開発背景・課題

大阪から宮崎に移住し農業に携わった経験から、作付け計画が「経験と勘」に依存していることを実感。連作障害・野菜の相性・施肥量など判断が難しく、計画に多くの時間と労力がかかる課題を、データとロジックで解決したいと考えた。

また、情報セキュリティマネジメント試験合格（2024年12月）の知識を活かし、セキュリティを意識した実装を行う。

---

## 3. ターゲットユーザー

| 区分 | 対象 |
|---|---|
| メイン | 市民農園・家庭菜園ユーザー（中級者） |
| サブ | 家庭菜園初心者・新規就農者 |
| その他 | 教育・食育目的の利用者 |

---

## 4. 競合調査

既存サービスは「記録・管理・経営分析」が中心。「作付け計画を視覚的に立てる」機能に特化したサービスは存在しない。

| サービス | ターゲット | 主な機能 | FarmPlanとの違い |
|---|---|---|---|
| Agrion（アグリオン） | プロ農家 | 農業日誌・販売管理・販路支援 | 作付け計画の自動化なし |
| アグリノート | プロ農家 | 営農支援・GAP認証・農機連携 | 家庭菜園・市民農園向けでない |
| AGRIHUB | プロ農家 | 農薬管理・売上分析 | 計画機能なし |
| 畑らく日記 | 家庭菜園 | 栽培記録 | 区画可視化・連作チェックなし |
| **FarmPlan（本サービス）** | **市民農園・家庭菜園** | **区画グリッド・連作チェック・シミュレーション** | **計画特化の空白領域を狙う** |

---

## 5. 機能要件

### 認証
- ユーザー登録・ログイン・ログアウト
- セッション管理

### 畑マップ（field.php）
- 畑をグリッド（1m²単位）で可視化
- 区画クリックで野菜の植え付け・編集・収穫・失敗・取り消しが可能
- 科ごとに色分け表示（6科）
- 連作障害チェック（AJAX）：過去3年の同科栽培を警告
- 株数・植え付け日の登録・編集

### シミュレーション（plan.php）
- 来季の作付け計画をグリッドで立てる
- 本番データ（actual）とは別に管理（mode: plan）
- サーバーサイドで全区画の連作警告を事前表示

### 栽培記録（history.php）
- 年・ステータス・科で絞り込み
- チェックボックスで複数記録を一括削除

### 設定（settings.php）
- プロフィール（ユーザー名・メール）変更
- パスワード変更
- 野菜マスタの追加・削除（品種登録対応）

### データ管理
- 畑の追加・削除（ON DELETE CASCADEで関連データも連動削除）
- 今年の記録リセット・全記録削除
- 畑名の重複防止（PHP + DB UNIQUE制約）

## 6. 非機能要件

### セキュリティ

| 対策 | 実装内容 |
|---|---|
| パスワード暗号化 | `password_hash()` / `password_verify()` |
| SQLインジェクション対策 | PDOプリペアドステートメント |
| XSS対策 | `htmlspecialchars()` による出力エスケープ |
| セッション固定攻撃対策 | ログイン後に `session_regenerate_id()` を実行 |
| 認証ガード | 全ページでセッション確認、未ログインはリダイレクト |
| 所有権確認 | 他ユーザーのデータへのアクセスをDBレベルで制限 |
| 機密情報管理 | `.env` ファイルで管理、`.gitignore` で除外 |

### ユーザビリティ
- モーダルでページ遷移なく操作完結
- 連作警告をリアルタイム（AJAX）で表示
- レスポンシブ対応

## 7. 画面構成・遷移

```
login.php / register.php
    ↓ ログイン成功
index.php（ホーム）
    ├── field.php（畑マップ）
    │     └── 区画クリック → モーダル（植え付け・編集・収穫・失敗・取り消し）
    ├── plan.php（シミュレーション）
    │     └── 区画クリック → モーダル（計画・削除）
    ├── history.php（栽培記録）
    └── settings.php（設定）
```

**画面遷移フロー**
1. シミュレーション → 畑マップで植え付け → status: growing
2. 畑マップ → 「収穫した」 → status: harvested → 栽培記録に保存
3. 栽培記録の蓄積 → 翌年のシミュレーション時に連作チェックへ自動反映

---

## 8. ステータス設計

`plot_seasons.status` カラムで栽培の状態を管理。

| ステータス | 意味 | 移行条件 | 画面 |
|---|---|---|---|
| `planned` | 計画済み | シミュレーションで配置 | シミュレーション |
| `growing` | 栽培中 | 畑マップで植え付け | 畑マップ |
| `harvested` | 収穫済み | 「収穫した」ボタン押下 | 畑マップ → 栽培記録 |
| `failed` | 失敗 | 「失敗した」ボタン押下 | 畑マップ → 栽培記録 |

`harvested` / `failed` のデータは翌年のシミュレーション時に連作チェックの履歴として参照される。

## 9. DB設計（7テーブル構成）

```
users           ユーザー管理
fields          畑の管理（サイズ・名前）
plots           区画（1m²単位、fieldsに紐づく）
plot_seasons    栽培履歴（連作チェックの核心テーブル）
vegetables      野菜マスタ（科・品種）
companion_rules コンパニオンプランツルール
harvests        収穫記録
```

**設計のポイント**
- `plot_seasons.mode`（plan / actual）でシミュレーションと実績を同一テーブルで管理
- `plot_seasons.quantity` で1区画あたりの株数を管理
- `fields(user_id, name)` にUNIQUE制約で畑名の重複を防止
- `fields` 削除時は `ON DELETE CASCADE` で `plots` → `plot_seasons` を連動削除
